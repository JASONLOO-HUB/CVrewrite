# CVrewrite

一个面向产品经理简历场景的 Skill：  
把 **目标 JD + 原始简历 + 项目代码仓库** 转成一段**贴岗、自然**，而不是照抄 JD 关键词的项目经历。

它不是普通的“简历润色器”。  
它的核心目标是：

- 先从项目材料和代码仓库里提炼你的做这个产品的**原始意图**
- 再结合你的简历和目标 JD 做**有限度包装**
- 最终产出一段**像产品经理简历**，而不是像技术方案、项目文档或关键词拼贴的项目经历

## 这个 Skill 解决什么问题

很多简历改写工具在处理项目经历时，容易出现两个极端：

1. 只盯着 JD 写，导致项目原本的产品意图被打乱，最后变成四不像
2. 只盯着代码和实现写，导致成品太技术、太像开发说明，不像产品经理简历

`CVrewrite` 的处理顺序是反过来的：

1. 优先读取 PRD、设计文档、计划文档等材料，提炼项目原始意图
2. 如果这些材料不完整，就以代码仓库为主要上下文补足理解
3. 再结合原简历和目标 JD，判断这个项目最适合补哪 1-2 个岗位点
4. 最后用产品经理语言写出一段简历项目经历

## 适用场景

适合这些情况：

- 你正在投产品经理相关岗位
- 你手头有一份目标 JD
- 你有自己的原始简历
- 你还有一个项目代码仓库，里面可能包含：
  - PRD / 设计文档 / 计划文档
  - 代码实现
  - worklog / 需求笔记

不适合这些情况：

- 软件工程师简历
- 纯粹想写一篇项目复盘
- 没有仓库，只想做一句话润色

## 设计原则

这个 Skill 的核心约束有五条：

1. 原始意图优先于 JD
2. 计划中的设计按已实现处理，可写进简历
3. 一段经历通常只包装 1-2 个最强匹配点，不追求覆盖整份 JD
4. 最终输出必须像产品经理简历，而不是像技术方案或设计文档
5. 默认使用通用产品经理语义，只有行业类型明确时才进一步细化

## 为什么只包装 1-2 个 JD 匹配点

简历里的一段项目经历，最好读起来像**一个清晰的产品故事**，而不是把 JD 上的能力词挨个打卡。

- **避免过度对齐 JD**：一段经历若试图覆盖整份 JD，常见后果是 bullet 像在「对照 JD 填空」，项目定位反而模糊，最后既不像真实项目，也不像可信的产品经理叙述。Skill 把 JD 当成**选台前的辅助**，而不是改写项目的起点。
- **和输出形态一致**：默认产出是「一句概述 + 两条 bullet」，天然适合承载 **1-2 条最强主线**；再多方向容易挤在同一段里互相冲淡。
- **留给整份简历分工**：其他 JD 要点可以由别的经历、别的项目来体现；这一段只负责**补位**，而不是重复你简历里已经写得很满的能力。

更细的「对齐过头」的典型表现与顺序要求见 `[references/jd-selection.md](references/jd-selection.md)`。

## 这两个匹配点是怎么选的

选点顺序在 Skill 里是固定的：**先简历与岗位画像，再项目能自然支撑什么**。

1. **看候选人画像（结合原简历）**
  你已经在哪些能力上写得很强？还有哪些岗位关键点**没被清楚体现**？  
   目标往往是让这个项目去**补缺口**，而不是重复已有强项。
2. **看项目的自然匹配点（结合仓库与材料）**
  在「原始意图已经清楚」的前提下，只保留**同时满足**下面条件的点：  
  - 和项目原始意图一致  
  - 材料与仓库上下文**确实能支撑**（不硬编）  
  - 对目标岗位有价值  
  - 在整份简历里具有**补位意义**
3. **在候选集合里只留最强的 1-2 个**
  例如复杂系统抽象、核心场景与关键机制、规划与迭代、商业化或业务闭环、用户路径等——以项目**真实最强叙事**为准；若项目最强的是「系统抽象 + 关键机制」，就不会为了贴 JD 再硬塞「跨团队」「商业分析」等若即若离的标签。

输出里的「JD 选点与理由」一层，会把**选了哪 1-2 个点、为什么、排除了哪些**简要写清楚，方便你自查是否贴岗且不失真。

## 输入与输出

### 输入

- 目标 JD
- 用户当前简历
- 项目代码仓库

### 输出

默认会组织成四层：

1. 仓库上下文摘要
2. 原始意图提炼
3. JD 选点与理由
4. 最终项目经历成稿

## 工作流

这个 Skill 的标准流程如下：

1. 检查输入是否齐全
2. 扫描代码仓库，建立高价值上下文索引
3. 优先阅读 PRD / 设计 / 计划材料
4. 再阅读入口文件、核心目录和关键代码文件
5. 用 STAR 法则 + 产品业务周期视角提炼原始意图
6. 结合原简历和 JD，只选 1-2 个最匹配岗位点
7. 生成最终项目经历
8. 主动做一轮自审，检查是否太技术、太像文档、或为了贴 JD 而扭曲项目叙事

## 仓库扫描脚本

Skill 内置了一个轻量扫描脚本：

```bash
python3 scripts/scan_repo_context.py /path/to/repo
```

它不会直接帮你写简历，而是先输出高价值上下文索引，比如：

- 优先阅读的文档 / 计划 / 设计材料
- manifest / 入口配置
- 值得优先查看的代码文件
- 建议阅读顺序

如果 `jd.md` 或 `resume.md` 恰好也放在仓库里，可以排除掉，避免污染项目上下文：

```bash
python3 scripts/scan_repo_context.py /path/to/repo \
  --exclude /path/to/jd.md \
  --exclude /path/to/resume.md
```

## 输出风格

最终项目经历默认写成：

- 一句项目概述
- 两条 bullet

通常分工如下：

- 第一条：核心场景 / 系统抽象 / 产品判断
- 第二条：关键机制 / 业务闭环 / 商业化 / 留存 / 增长中的第二层价值

如果上下文不够稳定，Skill 不会硬写，而是先向用户做简短澄清提问。

## 如何使用

把这个 Skill 放进你的 Skills 目录后，在支持 Skills 的 agent 环境里调用它。

一个典型的请求会像这样：

```text
Use $CVrewrite to turn this JD, my current resume, and my project repository into a PM-style project experience entry.
```

或者中文：

```text
用 $CVrewrite，基于这个 JD、我的原简历和项目代码仓库，生成一段产品经理简历里的项目经历。
```

## 它和普通简历润色工具有什么不同

最大的不同有四点：

1. 它把**代码仓库**当作核心上下文，而不是只把它当“功能清单”
2. 它强调先提炼**原始意图**，再做 JD 包装
3. 它明确限制 JD 对齐力度，避免把项目写成关键词拼贴
4. 它默认使用通用产品经理语义，而不是预设某个行业模板

## 当前边界

这个 Skill 更擅长处理：

- 有真实仓库的项目
- 有产品设计意图的项目
- 需要从复杂上下文中提炼产品主线的项目

它不负责：

- 编造市场结果或业务数据
- 把不存在的功能写进简历
- 用一段经历覆盖所有 JD 要点
