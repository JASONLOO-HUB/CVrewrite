# 澄清提问阈值

## 用途

当你已经扫描仓库并读过高优先级材料后，用这份参考判断是否必须向用户提问。

## 原则

这个 skill 可以容忍信息不完整，但不能容忍原始意图判断失真。

所以阈值要稍微严格：

如果上下文不足以稳定提炼原始意图，就直接问，不要硬写。

## 必问情形

扫描和阅读之后，只要出现以下任意一种，就应直接澄清：

1. 仍然无法判断项目的核心体验是什么
2. 仍然无法判断项目服务的用户 / 场景是什么
3. 仍然无法判断项目更偏效率工具、内容产品、平台产品、交易协作、社区互动还是其他主线
4. 代码和文档给出的主线明显冲突，无法自行解释
5. 项目仓库太碎片化，看到的是零散实验而不是稳定主线

## 可问但不一定要问

下面这些问题只有在确实影响成稿时才问：

- 这段希望纯产品口径，还是允许少量偏技术表达
- 这段更想强调核心场景与关键机制，还是强调业务闭环 / 商业化 / 留存
- 是否有明确不想提的项目方向

## 提问规则

- 一次只问 1 个最关键问题
- 问题要短
- 先问会决定原始意图判断的问题，再问写法偏好

## 推荐提问

当原始意图不清楚时，优先问：

- `这个项目最初最想解决的核心问题是什么？`

如果需要进一步收敛，再问：

- `你希望这段更强调核心场景与关键机制，还是更强调业务闭环 / 长期价值？`

如果只是写法不明确，再问：

- `这段项目经历你希望保持纯产品口径，还是可以保留少量偏技术的产品表达？`
