# 技术上下文改写成产品经理语言

## 用途

当源材料主要来自代码仓库，或者文档很像设计说明、技术说明时，读这份参考，把内容翻译成产品经理简历语言。

## 核心目标

不是删除所有技术信息，而是把技术信息改写成：

- 产品定位
- 核心场景 / 核心体验
- 系统抽象
- 用户路径
- 关键机制
- 业务闭环
- 增长 / 留存 / 商业化等产品层逻辑

## 优先表达的层级

优先写：

- 项目定位
- 核心场景
- 核心体验
- 系统抽象
- 关键机制
- 业务闭环
- 冷启动 / 留存 / 长期成长设计

少写或改写后再写：

- 文件结构
- 模块清单
- 接口流程
- 服务拆分
- 数据库结构
- 纯技术架构描述

## 改写规则

### 规则 1：从“实现组成”改成“产品结构”

差的写法：

- 设计了首页模块、任务模块、消息模块、配置模块、支付模块

更好的写法：

- 将产品拆解为用户承接、核心使用场景、任务反馈与商业化闭环等核心系统

### 规则 2：从“技术职责”改成“产品边界”

差的写法：

- 设计了前后端职责拆分和奖励校验逻辑

更好的写法：

- 明确奖励发放、资产口径与用户激励之间的产品边界

### 规则 3：从“功能枚举”改成“更高一层的抽象”

差的写法：

- 设计了表单、筛选器、消息提醒、标签体系、引导页

更好的写法：

- 围绕核心场景设计关键机制、反馈机制与新手承接

### 规则 4：从“技术上下文”改成“业务周期”

差的写法：

- 规划登录、隐私授权、运营配置、埋点监测

更好的写法：

- 规划用户承接、关键转化节点、运营开关与数据监测等关键环节

### 规则 5：没有行业证据时，使用通用产品语言

如果仓库和项目材料没有明确说明产品类型，就默认使用通用产品经理语义：

- 用“核心场景”替代“玩法”
- 用“关键机制”替代行业专属术语
- 用“业务闭环”替代“经营闭环”

不要因为某些结构相似，就先入为主地把项目写成游戏、AI 工具或内容平台。

## 什么时候可以保留少量偏技术表达

仅在同时满足下面两点时保留：

1. 这个表达能体现产品判断，不只是实现过程
2. 用户没有明确要求“纯产品口径”

通常还能保留的词：

- 资产口径
- 奖励边界
- 数据监测
- 运营配置

通常应继续改写或删除的词：

- 前端负责什么
- 后端负责什么
- 接口如何调用
- 服务如何拆分
- 数据库存什么

## 常用动词

优先使用：

- 主导
- 负责
- 设计
- 定义
- 拆解
- 抽象
- 规划
- 建立
- 推动

慎用：

- 开发
- 实现
- 编写
- 设计数据库
- 搭建接口
