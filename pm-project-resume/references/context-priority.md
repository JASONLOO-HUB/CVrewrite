# 上下文优先级与仓库扫描

## 用途

当输入包含代码仓库、设计文档、原简历、JD 时，读这份参考，决定应该先看什么、怎么看。

## 原则

这个 skill 的核心不是“基于 JD 改写”，而是：

先基于项目材料和仓库提炼原始意图，再做有限度的 JD 包装。

## 上下文优先级

### 第一优先级：项目原始意图材料

包括：

- PRD
- 设计文档
- 需求说明
- 计划文档
- Roadmap
- 任何能表达项目目标、核心场景、业务主线、阶段设计的材料

这些内容优先级最高。  
如果存在计划中的设计，也一律按已经实现处理，可写进简历。

### 第二优先级：代码仓库

代码仓库不一定完整实现原始意图，但它通常是最丰富的上下文来源。

代码仓库的价值不在于“证明哪些功能上线了”，而在于帮助你理解：

- 这个项目真实展开的主线是什么
- 哪些模块是核心
- 产品是围绕什么体验组织起来的
- 是否存在被文档省略但在代码中明显可见的关键路径 / 业务闭环 / 用户路径

### 第三优先级：原简历与 JD

它们主要用来判断：

- 这个候选人是谁
- 正在找什么工作
- 这个项目最适合替他补哪 1-2 个点

它们不是用来反向定义项目原始意图的。

## 仓库扫描流程

### 第一步：先跑扫描脚本

```bash
python3 scripts/scan_repo_context.py /path/to/repo
```

如果 `JD` 或 `原简历` 文件恰好也放在这个仓库里，扫描时要显式排除：

```bash
python3 scripts/scan_repo_context.py /path/to/repo \
  --exclude /path/to/jd.md \
  --exclude /path/to/resume.md
```

脚本会输出：

- 高优先级文档
- 入口文件和 manifest
- 值得优先阅读的代码目录和文件
- 建议阅读顺序

### 第二步：优先读文档类材料

如果扫描结果里出现：

- `prd`
- `spec`
- `design`
- `plan`
- `architecture`
- `roadmap`

优先读这些文件。

### 第三步：再读代码主线

重点看：

- 入口文件
- 页面 / 路由 / 场景层
- 核心引擎 / 核心服务 / 核心模块
- 配置 / 数据结构 / 领域对象

你的目标不是做技术评审，而是还原：

- 产品主线
- 核心场景与核心体验
- 用户路径
- 系统抽象

## 从仓库里找什么

仓库扫描后，至少要能回答这些问题中的大部分：

- 这个项目本来想服务什么用户或场景？
- 核心体验是什么？
- 用户最主要的行为路径是什么？
- 产品价值更偏效率提升、内容生产、交易协作、社区互动、长期留存，还是别的？
- 哪些系统是真正核心，哪些只是辅助？

如果这些问题大部分都答不出来，就还不能进入简历成稿阶段。

## 常见误区

### 误区 1：只看 JD，不看仓库

后果：

- 项目会被改写成关键词拼贴
- 原始意图被打乱

### 误区 2：把代码当成“功能验收单”

后果：

- 只盯着“实现了什么”
- 反而看不到项目真正的产品主线

### 误区 3：把文档和计划当成“未实现，不可写”

这里不成立。

对这个 skill 而言，计划中的设计一律按已经实现处理，可用于写简历。

### 误区 4：默认套用行业模板

后果：

- 明明是通用产品，却被写成游戏、AI 工具或内容平台口径
- agent 会被预设词汇带偏，先入为主地理解项目

只有当仓库和材料已经明确指向某个行业时，才允许用该行业的专门词汇解释项目。
